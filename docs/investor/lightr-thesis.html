Definição

01O que é Lightr

Lightr é um runtime de containers nativo em conteúdo (CAS), sem daemon e crash-only. É a peça que pega uma imagem e a coloca para rodar — o mesmo papel do containerd+runc por baixo do Docker — mas com uma arquitetura fundamentalmente diferente.

CAS-nativeConteúdo endereçado por hash

Cada blob (camada, arquivo, artefato de build) existe uma única vez, identificado pelo seu hash. Deduplicação não é uma otimização — é a estrutura de dados. O que já está no nó nunca é baixado, extraído ou armazenado de novo.

DaemonlessO listener não é dono de nada

Não há um processo central de longa vida segurando o estado. O kernel e o disco são a fonte da verdade; o runtime é apenas uma visão sobre eles. Sem daemon, sem RAM presa só para "existir".

Crash-onlyRecuperar = reabrir a visão

Projetado para morrer a qualquer instante. kill -9 no meio de uma operação e nada se perde, nada precisa reconciliar — porque nunca houve um espelho em memória para desincronizar.

MemoizávelO mesmo trabalho nunca se repete

Como tudo é endereçado por conteúdo, operações (build, layer, exec) são memoizáveis pelo hash da entrada. Refazer trabalho idêntico é, por construção, um cache-hit.

Origem: Lightr é "o Docker do HuGR" — o runtime que herda as semânticas de CAS do CoreLink. O produto local (R0–R4) já está entregue.

Meu projeto neste ciclo

02lightr-cri — a porta para o Kubernetes

Lightr roda containers. Mas o mundo orquestra containers com Kubernetes. Para o Kubernetes usar o Lightr, ele precisa falar o protocolo padrão que o cluster espera: o CRI (Container Runtime Interface). lightr-cri é exatamente essa camada.

É a casca voltada ao kubelet: recebe as chamadas gRPC do Kubernetes (criar pod, rodar container, exec, logs, rede) e as traduz para o Lightr — sem manter estado próprio. Fiel ao princípio: mate o processo no meio de uma operação e nada se perde. É o que torna o Lightr plugável em qualquer cluster Kubernetes, em vez de ser só um runtime local.

kubelet  →  CRI gRPC  →  lightr-cri (stateless)  →  Lightr / CoreLink CAS

Construído em repo próprio, contra um contrato de seam congelado — integra ao HuGR no fim por troca de contrato, nunca copy-paste. Conformidade é a suíte de aceitação (red→green).

03O que é melhor — e o que muda

A barra que perseguimos é explícita e honesta: dominância estrutural onde a arquitetura nos dá vantagem por construção (pull, footprint, restart, memoização) e paridade no piso de kernel — onde todo runtime, no fim, faz a mesma syscall.

Plano de prova

05Como vamos provar que é melhor

O método é um head-to-head reprodutível contra containerd/Docker, no mesmo host, com as mesmas imagens — e a correção vem antes da velocidade.

  1. Piso de correção primeiro. Antes de qualquer claim de performance, o Lightr passa na mesma suíte oficial que valida os concorrentes (critest). Já feito: 33/33 verde. Sem isso, "mais rápido" é irrelevante.
  2. Bancada A/B idêntica. Mesmo runner Linux, mesmas imagens OCI, mesmo kernel/cgroups. Lightr vs. containerd lado a lado — única variável é o runtime.
  3. Probes deterministas. Cada KPI estrutural medido por uma sonda reprodutível: bytes transferidos no pull, RSS residente ocioso, tempo de cold start (cache frio vs. quente), tempo de recuperação pós-kill -9, bytes em disco para N imagens.
  4. Assinatura por CI (fail-closed). O número só é publicado depois que um run o assina — registrado com commit, host e versões. Sem run, sem claim. É o mesmo princípio que já governa a conformidade.
  5. Regressão travada (GREENLIST). Cada vitória vira gate: o que ficou verde não pode regredir sem o pipeline acusar. KPIs não são fotos — são contratos contínuos.

Em uma frase: provamos como provamos a conformidade — com uma suíte que ou está verde, ou falha alto. A diferença é que agora a métrica é desempenho estrutural, não só correção.
