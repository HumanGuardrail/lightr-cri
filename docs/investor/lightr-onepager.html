O problema

O stack de containers de hoje carrega peso morto.

Todo cluster Kubernetes hoje roda sobre um daemon pesado e cheio de estado. Ele duplica armazenamento, demora a subir, ocupa muita memória, e quando algo desincroniza, entra num loop de reconciliação. É a camada mais crítica da infraestrutura — e a mais frágil.

Docker / containerd hoje

  • Daemon stateful — um processo dono de tudo; se ele cai, o nó sofre.
  • Armazenamento redundante — camadas repetidas, pull lento, disco desperdiçado.
  • Cold start lento — descompactar e montar antes de rodar.
  • Reconciliação — estado espelhado que desincroniza e precisa se "curar".
  • Footprint alto — memória e CPU presos só para manter o daemon vivo.

Lightr

  • Sem daemon — o listener não é dono de nada; o kernel e o disco são a verdade.
  • CAS-native — conteúdo endereçado por hash: deduplicado por construção, pull mínimo.
  • Cold start estrutural — sem cópia: o conteúdo já está pronto no CAS.
  • Crash-only — kill -9 a qualquer momento; nada se perde, nada reconcilia.
  • Footprint mínimo — zero estado próprio para manter de pé.
Por que é melhor — e não é marketing

Quatro vantagens estruturais, não ajustes.

Estas vantagens vêm do design, não de otimização. É a diferença entre "mais rápido" e "rápido por construção".

📥

Pull

Conteúdo endereçado por hash (CAS): cada blob existe uma vez. Imagens compartilham camadas de verdade — o que já está no nó nunca é baixado de novo.

🪶

Footprint

Sem daemon, sem estado espelhado. O runtime é uma visão sobre o disco e o kernel — não uma cópia que precisa de RAM para existir.

♻️

Restart

Crash-only por princípio. Reiniciar é instantâneo porque não há nada para reconciliar — a fonte da verdade nunca esteve em memória.

🧠

Memoização

Operações são endereçadas por conteúdo: o mesmo trabalho nunca é refeito. Build, layer, exec — tudo memoizável pelo hash.

O produto

Lightr corre por baixo. lightr-cri conecta no Kubernetes.

O Kubernetes fala com o runtime através de um contrato padrão: o CRI (Container Runtime Interface). Construímos o lightr-cri — uma camada stateless que deixa o kubelet dirigir o Lightr diretamente. Plugue no cluster e ele simplesmente funciona, como conformidade comprovada.

kubelet → CRI gRPC (unix socket) → lightr-cri
shell stateless → CoreLink CAS
via Lightr
O listener não é dono de nada. Mate-o no meio de uma operação — nada se perde, nada reconcilia, porque nunca houve um espelho.
